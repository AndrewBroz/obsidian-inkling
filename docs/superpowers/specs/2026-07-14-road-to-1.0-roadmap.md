# Road to 1.0 — Roadmap

**Status:** approved 2026-07-14. This document is the map; each phase gets its own spec.

Inkling passes its happy paths and looks right. It is not trustworthy. An investigation on
2026-07-14 (four parallel code audits) found that the edit engine silently relocates keystrokes,
that Suggest mode does not track every edit, and that comment threads can rebuild themselves
differently on different keystrokes. Those are 1.0 blockers in a way that pixels are not.

This roadmap sequences the work by **what survives the parser fork**, so that nothing is built twice.

---

## The through-line: one bug, three costumes

Every position predicate in the codebase is **closed on both ends**. `cursor_inside(p)` is
`from <= p && p <= to`. The `@flatten-js/interval-tree` backing `at_cursor`, `ranges_in_interval`
and `contains_range` reports two intervals that merely *touch* as *intersecting*
(`not_intersect` uses strict `<`).

So the codebase cannot distinguish **"my cursor is beside this range"** from
**"my cursor is inside this range."** Those are different questions with different right answers.
Three separate defects are the same missing word:

| Symptom | Where |
|---|---|
| Typing before a highlight teleports the character past it | `mark.ts:492-503` |
| Comment-draft eligibility flips between open and commit | `add-comment.ts:41` vs `:202` |
| Comment threads duplicate their replies on edit | `range-state.ts:143` |

`add-comment.ts` is the proof: it needed the strict answer, could not get it, and hand-rolled one
inline with a comment explaining why the shared predicate was wrong.

## Phases

### Phase 1 — Edit-engine integrity (NEXT)
Spec: `2026-07-14-edit-engine-integrity-design.md`

Everything here is orthogonal to the parser and would need doing under any range model.

1. Honest position predicates; the teleport and the thread duplication die with them.
2. Suggest mode fails **closed** (denylist, not allowlist) — it currently lets unrecognised edits
   through untracked.
3. Partial coverage of a pending addition **retracts** instead of folding (`reject_all` currently
   resurrects text that was never in the document).
4. Typing strictly inside a highlight **splits** it rather than teleporting out of it.
5. Two UI bugs: pill click focuses the draft box; the draft card's `offsetTop` crash is guarded.

**Ships as a release before Phase 2 begins.** The bleeding is live.

### Phase 2 — Fork the parser; overlap dialect
Spec: not yet written.

**Decided:**
- **Fork.** We currently cannot read the parser we ship. `@fevol/lang-criticmarkup@2.1.0` publishes
  only a compiled 3KB Lezer table; its source repo (`fevol-forks/lang-criticmarkup`) is 404. The
  public ancestor (kometenstaub, 2022) is ~60 lines. Zettlr's fork is an active reference.
- **Overlap is supported via an ID dialect.** CriticMarkup structurally cannot express overlap:
  `{==a {==b==} c==}` closes the outer range at the inner `==}`. An ID-tagged dialect can:
  `{==#a1 The quick {==#b2 brown==#a1} fox==#b2}`.
- **The dialect appears only when needed.** Plain CriticMarkup is written always; an ID pair is
  emitted *only* at the moment a range actually overlaps another. A note that never overlaps stays
  byte-identical to standard CriticMarkup and portable to Zettlr/pandoc. You leave the standard only
  when you ask for something the standard cannot say.

**Architectural consequence — the grammar stops parsing ranges.** Lezer is LR, and an LR grammar
has a finite set of nonterminals, so it *cannot* enforce "this `==#a1}` closes *that* `{==#a1`" —
ID equality is context-sensitive. Therefore:

```
today:   grammar produces RANGES  →  cursorGenerateRanges walks them  →  plugin logic
fork:    grammar produces TOKENS  →  a pairing resolver we own        →  plugin logic
```

The grammar gets *simpler* (emit `OpenHighlight(id)` / `CloseHighlight(id)` / text; no bracket
matching). Range construction moves into TypeScript — where it already effectively lives, and where
it can be tested.

**Known consequence of "only when needed":** thread membership still cannot use IDs in the common
case, because the common case has no IDs. Adjacency-threading (`right_adjacent`) therefore **stays**
— it is the CriticMarkup convention and it is what keeps `{==x==}{>>note<<}` portable. Phase 1 must
fix thread duplication properly; it will not be dissolved by the fork.

### Phase 3 — Comment coherence
Spec: not yet written.

There are **four** comment surfaces, sharing no renderer, no menu, and no concept of a thread:

| Surface | Renders | Reply UI | Per-item menu |
|---|---|---|---|
| Live-preview icon + hover tooltip | bare comments **only** | via context menu | yes |
| Gutter thread card | bare and anchored | inline reply box | thread-level only |
| Sidebar Annotations View | bare and anchored | none | yes |
| Reading-view post-processor | **every comment separately** | none | none |

- The hover tooltip renders *only* bare comments — not by design, but because
  `markup-renderer.ts:173` gates the icon on `range.base_range === range`, and an anchored comment's
  base is its highlight. It is the richest comment UI we have and it is unreachable for the comments
  users actually make.
- Reading view has no concept of a thread at all: `rangePostProcess` runs **per range**, so a
  3-reply thread draws three separate icons.
- "Bare" is not a flag; it is derived as `attached_comment === null`. A floating note and the head of
  an unanchored thread are therefore *the same boolean everywhere*. If they should look different,
  that distinction must be created.
- Docs-reference gaps still open: the connector line from marked text to its card; per-reply `⋮` menus.

## Explicitly out of scope for 1.0

- Out-of-band / sidecar comment anchors (considered and rejected: anchors drift, comments stop being
  portable CriticMarkup).
- Always-on IDs (considered and rejected: makes every note non-standard immediately).
